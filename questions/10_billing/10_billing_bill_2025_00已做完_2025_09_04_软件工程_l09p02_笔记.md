# q
在交互式变基（rebase）暂停状态下，`git status` 显示了哪些关键信息？
# a
`git status` 会显示当前分支名称后带有 `|REBASE` 标记，列出剩余待执行的命令（如 `pick`），并提示当前正处于“编辑某个提交”的阶段，同时给出后续操作建议：
- 使用 `git commit --amend` 修改当前提交
- 使用 `git rebase --continue` 完成修改后继续变基

# q
`git reset <commit-hash> --hard` 命令的作用是什么？请结合示例说明。
# a
该命令强制将 `HEAD` 指针移动到指定的提交，并重置暂存区和工作目录，使其与该提交完全一致，**所有未提交的更改都将被丢弃**。  
示例中：
- `git reset 6d7a018c --hard` 使 `HEAD` 回到提交 `6d7a018`，丢弃了后来创建的 `modify setup` 提交
- `git reset c45c8e3309 --hard` 又将 `HEAD` 移回包含 `modify setup` 的提交 `c45c8e3`

# q
在变基过程中，如果想恢复到之前的某个提交状态，可以怎么做？
# a
可以直接使用 `git reset <commit-hash> --hard` 强制回退到任意提交。例如在变基暂停时，先通过 `git log` 查看提交历史，然后使用 `git reset 6d7a018c --hard` 就可以丢弃当前变基途中产生的修改，将工作目录恢复到该提交时的干净状态。

# q
示例中 `python setup.py` 执行后完成了哪些操作？
# a
脚本自动完成了以下操作：
1. 从 `main` 分支创建多个特性分支（如 `feature/new_feature_A` 到 `feature/new_feature_G`）
2. 在每个分支上添加或修改文件（如 `databaseInterface.c`、`loadEntryFile.c`）并生成对应的提交
3. 每完成一个分支的操作后自动切回 `main` 分支，确保最后停留在 `main` 上
