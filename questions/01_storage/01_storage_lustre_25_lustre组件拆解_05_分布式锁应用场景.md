# q
什么是电商高并发秒杀场景下的库存超卖问题？它是如何发生的？
# a
库存超卖问题指多个用户同时下单时，因并发查询和扣减库存操作不是原子性的，导致实际扣减库存总量超过真实库存。例如：用户A和用户B同时查询到库存为B，各自判断需求量A ≤ B后下单扣减，导致库存被重复扣减，最终库存变为负数，即超卖。问题根源在于“查询库存→判断库存→扣减库存”这三步操作未加锁，存在竞态条件。

# q
使用分布式锁解决库存超卖的核心原理是什么？其缺陷是什么？
# a
核心原理：通过对商品库存加分布式锁，使同一时刻只有一个请求能执行“查库存→判断→扣减”的逻辑，从而保证数据准确性，避免超卖。  
缺陷：分布式锁导致对同一商品的请求串行化处理，并发能力弱，无法支撑高并发秒杀场景（例如同一商品瞬时大量下单），只适用于低并发（每秒不足10个请求）的普通电商场景。

# q
如何通过分段加锁方案解决分布式锁并发能力弱的问题？
# a
将商品总库存切分为多个分段（例如1000件库存拆成20个段，每段50件），每个分段用独立的锁控制。请求到来时通过随机算法选择一个分段库存，对该分段加锁后执行查库存、判断、扣减操作。这样最多可同时处理分段数量的并发请求，将串行改为分段并行，大幅提升吞吐量（例如20个分段可达到每秒处理1000个请求的理论上限）。

# q
在分段加锁方案中，当某个分段库存不足时应如何处理？
# a
如果请求对选中的分段加锁后发现该分段库存不足，必须自动释放锁，并立即选择下一个分段库存（换一个分段）再次尝试加锁和处理。这个过程需要实现自动重试切换，以保证请求最终能被满足，同时避免在库存不足的分段上长时间阻塞。
