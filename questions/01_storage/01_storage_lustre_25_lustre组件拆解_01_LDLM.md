# q
LDLM是什么？它的两个主要作用是什么？
# a
LDLM（Lustre Distributed Lock Manager）是 Lustre 的分布式锁管理器，基于 VAX DLM 实现。其两个主要作用是：
1. 提供锁定机制，确保所有 Lustre 节点上的数据一致性。
2. 允许客户端缓存由锁保护的状态，直到有冲突的锁被请求或因 LRU 策略过期。

# q
LDLM 命名空间包含哪两种类型？它们各自的特点是什么？
# a
LDLM 命名空间包含服务器命名空间和客户端命名空间。
- **服务器命名空间**：了解所有锁，具有权威性，可决定锁的授予与冲突处理。
- **客户端命名空间**：仅了解该客户端自己持有的锁。
每个 Lustre 服务在服务器端有一个服务器命名空间，每个客户端连接该服务时创建对应的客户端命名空间。客户端获取的每个锁由服务器端和客户端的两把内存锁组成，通过 cookie 链接，这类锁称为远程锁；仅服务器持有且无客户端引用的锁称为本地锁。

# q
LDLM 分布式锁支持哪六种模式？各自的典型使用场景是什么？
# a
六种锁模式及典型场景：
- **EX（独占）**：MDS 创建新文件前请求父目录锁。
- **PW（保护写）**：客户端向 OST 请求写锁。
- **PR（保护读）**：客户端从 OST 请求读或打开文件执行时授予。
- **CW（并发写）**：文件打开操作期间，MDS 授予客户端的写锁。
- **CR（并发读）**：路径查找时，MDS 在中间路径组件上授予 inodebit 锁。
- **NL（空模式）**：无锁定要求。

# q
根据锁兼容性矩阵，EX 模式可以和哪些模式共存？
# a
EX 模式仅与 NL（空模式）兼容，与 CR、CW、PR、PW 均不兼容。兼容矩阵如下：
```
      NL  CR  CW  PR  PW  EX
 NL    1   1   1   1   1   1
 CR    1   1   1   1   1   0
 CW    1   1   1   0   0   0
 PR    1   1   0   1   0   0
 PW    1   1   0   0   0   0
 EX    1   0   0   0   0   0
```
