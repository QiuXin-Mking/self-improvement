# q
Goofys 挂载 S3 存储桶时，如何解决目录下文件名或内容中的中文字符显示问题？
# a
需要在挂载时设置合适的字符编码或区域设置。在挂载 Goofys 时，通过 `--dir-mode` 和 `--file-mode` 不影响编码，主要应在容器或系统环境中设置 `LANG` 和 `LC_ALL` 环境变量为支持 UTF-8 的值（如 `en_US.UTF-8` 或 `zh_CN.UTF-8`），确保文件系统层能正确处理中文。也可以检查 S3 对象键本身是否为 UTF-8 编码。

# q
Goofys 是否必须使用特权模式（`--privileged`）挂载？如何避免特权模式同时保证 GPU 可见？
# a
Goofys 通常不需要特权模式，它基于 FUSE 用户态文件系统。要在非特权容器中运行 Goofys，需要确保容器具有 `SYS_ADMIN` 能力并挂载 `/dev/fuse` 设备，或通过使用 `--device /dev/fuse` 并添加 `--cap-add SYS_ADMIN` 来实现 FUSE 挂载。GPU 可见性通常依赖于 NVIDIA 容器运行时和 `--gpus` 参数，与挂载方式无关，只需正确安装 GPU 驱动和运行时即可，无需特权模式。

# q
在容器挂载的 Goofys 目录中使用 `vi` 修改文件时，为什么会导致整个挂载失效？
# a
Goofys 不支持原地修改（in-place modification），它并不是一个完整的 POSIX 文件系统。当使用 `vi` 等编辑器保存文件时，编辑器通常会创建临时文件并重命名，或者尝试对文件进行截断与重写，这会产生 S3 不支持的随机写入或重命名操作，导致操作失败并使挂载进入异常状态，表现为挂载点失效或 I/O 错误。解决方法：避免在 Goofys 挂载点内直接编辑文件，可复制到本地修改后再上传覆盖。
