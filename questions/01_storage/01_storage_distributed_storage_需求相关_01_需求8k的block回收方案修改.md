# q
`delete_dsid` 方法如何根据依赖关系确定可删除的 DS 列表？
# a
`delete_dsid` 从指定 `dsid` 出发，沿依赖链查找所有可删除的 DS，规则如下：
- 只有被该 DS 单线依赖的 DS 才能被连带删除，共享依赖的 DS 仅释放当前路径上的 DS。
1. 单链依赖（DS3→DS2→DS1）：若 DS3 可删，则释放列表为 `[DS3, DS2, DS1]`。
2. 分支中有共享父节点（DS5→DS4→DS1, DS2→DS1）：若 DS5 可删，释放列表为 `[DS5, DS4]`，DS1 因为被 DS2 依赖而保留。
3. 不同路径共享中间节点（DS5→DS2→DS1, DS3→DS2）：若 DS5 可删，释放列表为 `[DS5]`，DS2 被 DS3 依赖保留。
4. 快照同理，若快照卷依赖其他 DS，只释放被快照独占的路径。

# q
在 DS 的依赖关系图中，什么情况下会出现“不修改 ds 策略，此种情况不出现”的非法情况？
# a
当一棵树中一个叶子是源卷、其他叶子是视图卷，且存在某个 DS 被多个下游依赖，若删除操作可能违背快照或源卷的依赖约定（例如试图删除被多个快照共享的 DS 或源卷），这种依赖拓扑在设计上被视为非法，不修改策略就不会出现。

# q
8K 分支的 segment 回收流程中，Go 函数 `MergeDs` 承担什么核心作用？
# a
`MergeDs` 负责分析 DS 合并关系，找出可以安全回收的 segment ID 列表（例如 segid 10、12），并在复杂依赖中确定合并后的可释放段，供后续 `sioSegRecycle` 下发释放指令。

# q
segment 回收的异步处理中，脏块（dirty block）如何影响回收流程？
# a
通过 `ocache_query_blks_dirty` 查询 segment 的 `dirty_bitmap` 大小，若 bitmap 不为空则需要先将脏数据落盘；回收时需根据 bitmap 判断哪些 block 不可直接释放，避免数据丢失。
