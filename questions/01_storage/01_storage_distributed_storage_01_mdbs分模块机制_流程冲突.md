# q
Segment操作中流程冲突的本质是什么？
# a
冲突本质是多个并发操作试图修改相同的元数据（如 Segment 的 block_map 状态、元数据记录），如果不协调会导致状态不一致或 MDLOG 缺失。常见的冲突场景包括同时更新 block_map（持久化设为 PERSIST，压缩设为 Compact，分配回收设为 Normal），或者压缩期间缺少其他操作先行写入的 MDLOG。

# q
当 Segment 元数据不在内存中时，各流程如何处理？
# a
需要从持久化位置读取数据，还原内存中的元数据，然后再继续执行当前操作。

# q
元数据状态为压缩时，后续流程必须执行什么步骤？
# a
必须将元数据重写到 MDLOG，完成后将状态更新为 Normal，以保证后续操作能获得一致、完整的元数据。

# q
为什么压缩流程需要等待回收或删除操作完成？
# a
回收或删除操作会先写 MDLOG 再更新元数据，如果压缩在这些 MDLOG 写入完成前就生成压缩区，会导致压缩区缺失对应的回收/删除 MDLOG，破坏数据一致性。
