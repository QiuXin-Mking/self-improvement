# q
主程序使用 time.Sleep 等待协程时，协程输出不完整（如缺少最后一个数字）的典型根因是什么？
# a
主协程的睡眠时间短于协程执行总时间，导致主程序提前退出，协程被强制终止。示例中 `time.Sleep(2 * time.Second)` 而协程每 500ms 打印一个数字，5 个数字共需 2.5 秒，因此只输出 4 个数字后主程序结束，协程未能打印第 5 个数字。

# q
如何根据日志输出判断协程执行是否因主程序提前退出而中断？
# a
检查协程打印的序列是否完整，以及主程序结束信息是否在序列完成前出现。例如输出 `1 2 3 4 主程序结束` 表明协程在输出第 5 个数字前被终止，此时主程序 `fmt.Println("主程序结束")` 已执行，说明等待时间不足。

# q
解决 Go 协程因主程序提前退出而中断的标准流程是什么？
# a
使用 `sync.WaitGroup` 替代 `time.Sleep` 进行协程同步。

```go
var wg sync.WaitGroup
wg.Add(1)
go func() {
    defer wg.Done()
    // 协程逻辑
}()
wg.Wait()
```

`wg.Add(1)` 在启动协程前增加计数，协程完成时 `wg.Done()` 减少计数，主程序 `wg.Wait()` 阻塞直到计数归零，确保所有协程执行完毕，避免因时间估计错误导致协程提前终止。
