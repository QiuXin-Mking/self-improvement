# q
epoll 与 select/poll 的核心区别是什么？
# a
select/poll 每次调用都会线性扫描全部文件描述符集合，即使只有少量 socket 活跃，效率随集合增大线性下降。而 epoll 基于内核回调机制，仅对“活跃”的 socket 进行操作——当 fd 就绪时通过 callback 通知 epoll，有效避免无效遍历，实现类似 AIO 的伪异步。同时 poll 取消了 select 的最大描述符数限制（默认 1024），但没有解决遍历开销问题。

# q
select 系统调用的工作流程是怎样的？它存在哪些主要性能问题？
# a
select 的流程：
1. 进程调用 select 时，内核将其加入所有监控 socket 的等待队列；
2. 若已有 socket 就绪则直接返回，否则阻塞等待；
3. 当网卡收到数据触发中断，中断程序将数据写入 socket 接收缓冲区，并将等待队列中的进程移回工作队列（唤醒）；
4. select 返回后，程序仍需遍历整个 fd_set 才能确定哪些 socket 就绪。

主要性能问题：
- 每次调用需要将完整的 fd_set 从用户态复制到内核态，且每次阻塞/唤醒都涉及两次遍历所有 fd 的等待队列；
- 返回后应用程序必须 O(n) 遍历所有 fd 才能找到就绪的 socket；
- select 有最大描述符数量限制（通常 1024）。

# q
如何理解“阻塞”与“非阻塞”？它们和“同步/异步”的关系是什么？
# a
阻塞/非阻塞关注的是调用者在等待结果时的状态：
- 阻塞：调用后线程被挂起（进入 waiting 状态），直到结果就绪才返回；
- 非阻塞：调用立即返回，不等待结果，线程可以继续执行其他任务。

同步/异步关注的是调用者与通知结果的方式：
- 同步：调用者主动等待或轮询直到获取结果；
- 异步：调用者提交请求后立即返回，由被调用方通过回调等方式主动通知结果。

在处理 IO 时，阻塞和非阻塞通常都属于同步 IO（因为仍需自己调用 read/write 获取数据），只有使用特殊 API（如 IOCP）才算真正的异步 IO。
