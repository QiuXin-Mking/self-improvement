# q
Skill 文档由哪三个核心组件构成？它们分别承担什么角色？
# a
Skill 由三部分协同支撑治理与运行：
- **SKILL.md**：治理核心，包含元数据（名称、版本、作者）、行为指令、输入输出约束。
- **Resources**：辅助资产目录，建议按 `scripts/`（工具脚本）、`templates/`（报告模板）、`examples/`（样例数据）分类，SKILL.md 中需注明路径引用。
- **Tools**：可选的外部接口（数据库、API、本地脚本），用于扩展能力边界，复杂场景需明确调用方式与参数。

# q
SKILL.md 中应包含哪些元数据？版本号应采用什么格式？
# a
SKILL.md 必须包含的元数据有名称、版本、作者。版本号必须使用语义化版本格式，例如 `v1.0.0`。

# q
Skill 设计中的上下文策略包含哪三个关键部分？每个部分的核心内容是什么？
# a
上下文策略通过三层内容确保输出质量：
- **背景说明**：明确目标（如“自动生成合规周报，减人工80%时间”）、受众（如运营人员）及约束（如“仅中文输出”，安全约束优先）。
- **示例**：提供 1-2 个高质量输入→输出对照样例，并标注约束点；复杂技能可补充边界/异常样例。
- **验证**：定义输出格式约束（如 JSON Schema、Markdown 模板）和禁止内容（如禁止“大概”、禁止返回 API 密钥）。

# q
Skill 工具的三种常见集成方式分别是什么？
# a
1. **tools 字段绑定**：声明 API（URL、鉴权、参数）或脚本路径，敏感信息用环境变量注入，参数需校验 Schema。
2. **资源目录 + 命令行**：可执行文件放 `Resources/bin/`，在 SKILL.md 中声明命令（如 `python bin/analyze.py --input ${path}`），需确保跨平台适配及执行权限。
3. **日志记录**：记录输入（脱敏）、调用时间、输出状态，采用 JSON 格式便于审计，并定期归档（如 30 天）。

# q
在 Skill 版本管理中，主版本、次版本、修订号分别在什么情况下递增？
# a
- **主版本**：发生不兼容变更时递增。
- **次版本**：增加兼容性新功能时递增。
- **修订号**：进行 bug 修复时递增。
每次调整版本必须附带变更说明，例如 `[v1.1.0] 新增天气 API`。
