# q
Ceph 的 CRUSH 算法如何控制数据分布，故障域通常如何设计？
# a
CRUSH 算法通过 crushmap 定义存储拓扑结构，用规则将数据伪随机分布到 OSD 上。故障域设计通过在 crushmap 中设定分层结构（如 host、rack、row、room、datacenter），使同一数据的副本放置在不同故障域内，避免单点故障导致数据不可用。常见故障域层次：host（节点级） > rack（机架级） > row（行级）。

# q
Ceph 中一次写请求从客户端到 OSD 副本的完整数据路径是怎样的？
# a
写请求路径：客户端通过 librados 或 librbd 等接口找到目标 PG，根据 OSDMap 和 CRUSH 算法计算 PG 所在的主 OSD（Primary OSD）。客户端直接将数据写入主 OSD，主 OSD 负责将请求复制到从 OSD（Replica OSD），等待所有副本或多数副本写入确认后，返回 ACK 给客户端。涉及组件：Client → OSD primary → OSD replica(s)，再由主 OSD 汇聚应答。

# q
Ceph 如何通过 Scrubbing 机制保持数据完整性？light scrub 与 deep scrub 有何区别？
# a
Scrubbing 是 Ceph 的自动数据校验机制。Light scrub 仅校验对象元数据（大小、属性）的匹配性，不读取完整数据，速度较快。Deep scrub 则逐位比对所有副本的数据内容，确保无静默数据损坏（bit rot），但消耗大量 IO 和 CPU，通常调度在低负载时段。两种 scrub 均由 PG 的主 OSD 发起，通过比较副本间的校验和来检测不一致。

# q
OSD 出现 nearfull 状态时，若暂无物理扩容条件，有哪些应急处理方案？
# a
应急方案包括：降低 full/nearfull 告警阈值以争取时间；清理冗余数据、删除无用快照或回收空间；调整 Crush map 让数据分布更均匀；临时增加 OSD 节点的权重或使用低成本介质（如增加大容量 OSD 做临时填充）；启用压缩功能（如 BlueStore compression）减少磁盘占用；必要时手动触发全局 scrubbing 释放碎片空间；若为测试环境可暂时将集群写策略变为“满后停止”以外的更宽松模式（不推荐生产环境）。

# q
Ceph RADOS 的 PG 是如何实现故障恢复的？recovery 与 backfill 如何协调？
# a
当 OSD 加入或离开时，PG 的 epoch 版本发生变化，CRUSH 算法重新映射 PG 到 OSD。PG 经历 peering 状态协商一致后，根据 PG log 判断不同副本间的数据差异。recovery 优先于 backfill，主要恢复已知缺失的增量对象（根据 PG log 中缺失的 entry 恢复），backfill 则全量同步那些无历史记录的完整 PG 数据。通过参数如 `osd_max_backfills`、`osd_recovery_max_active` 控制并发的恢复/回填数量，避免冲击前端业务。
