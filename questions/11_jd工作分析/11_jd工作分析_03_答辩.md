# q
CRUSH 算法在 Ceph 中的作用是什么？它如何避免元数据单点瓶颈？
# a
CRUSH 是 Ceph 的数据放置算法，替代了传统元数据查表。给定 PG ID，它通过 crushmap 中的层级结构和 rule 伪随机计算出 PG 应落在哪些 OSD 上。这样客户端无需查询中心化元数据就能直接计算数据位置，避免了单点瓶颈。crushmap 定义 bucket 层级（root→rack→host→osd），rule 指定选择策略（take→choose→emit）。常用命令：`ceph osd crush tree` 查看结构；`crushtool` 离线编译/反编译。深圳地铁项目按 host 级故障域 3 副本部署，单台宿主机挂 12 个 OSD 未丢数据。

# q
PG 数量如何计算？PG 过多或过少会有什么问题？
# a
PG 数量计算公式：`(OSD 数量 × 100) / 副本数`，结果向上取 2 的幂。PG 过多会导致 OSD 内存和 CPU 压力大；过少会导致数据分布不均衡。PG 核心状态包括 up（物理位置）、acting（实际服务 IO）和 peering（数据一致性协商）。正常状态为 active+clean。常见异常状态：stale（OSD 未上报）、inconsistent（scrub 发现不一致）、degraded（副本不足）。查看命令：`ceph pg stat` / `ceph pg dump`。扩容时需控制新 OSD 的 crush weight，逐步调回并限制 backfill 速率。

# q
FileStore 为什么存在双写问题？BlueStore 如何解决双写？
# a
FileStore 的写路径是：数据先顺序写入 Journal（SSD 分区），后台再搬移到 XFS 上的数据文件，导致同一数据写两次（写放大 2x 以上），并伴随 XFS 目录碎片、filestore_merge 竞争和 LevelDB 元数据膨胀。BlueStore 去掉了文件系统层，直接管理裸盘。它将磁盘分为 block.wal（RocksDB WAL）、block.db（元数据）和 block（对象数据）三个区域。对象数据直接写入 block 分区，无二次搬运；元数据写入 RocksDB，其 WAL 仅保证元数据一致性。避免了双写和文件系统碎片。典型配置：NVMe SSD 分给多个 HDD OSD 做 block.db，HDD 做 block。迁移后延迟 p99 下降约 30%。

# q
Ceph 纠删码（EC）和多副本的适用场景与优缺点分别是什么？
# a
多副本（如 3 副本）可用容量为 33%，能容忍 2 个 OSD 故障，简单可靠，适合 RBD 块存储、CephFS 元数据、热数据等对延迟敏感的场景。EC 纠删码以 k 数据块 + m 校验块存储，如 4+2 可用容量 67%，容忍任意 2 块故障，节省存储成本，但读写需要编解码，CPU 开销和延迟高，适合大文件、冷数据、对象存储。EC 的 overwrite 需要 Read-Modify-Write，性能差。实际案例：RGW data pool 用 4+2 EC，index pool 用 3 副本，亿级对象下节省约 40% 存储成本。

# q
Ceph 集群中 MON 和 OSD 的主要职责分别是什么？维护奇数个 MON 的原因是什么？
# a
MON 负责维护集群元数据中心，包括 cluster map（OSDMap/PGMap/CRUSHMap/MDSMap），使用 Paxos 协议保证一致性，通常部署 3 或 5 个（必须奇数）。OSD 是存储数据的守护进程，每个磁盘一个 OSD，负责数据读写、复制、恢复和 scrub。MGR 提供监控与 dashboard，从 Luminous 起必须部署。RGW 提供 S3/Swift 兼容对象存储网关，MDS 管理 CephFS 元数据。奇数 MON 是为了避免脑裂并保证 quorum 选举。
