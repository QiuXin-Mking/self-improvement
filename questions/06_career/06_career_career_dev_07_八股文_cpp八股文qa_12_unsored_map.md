# q
hash冲突是如何触发的？
# a
当哈希函数将不同的键（如 "123" 和 "1234"）计算出相同的哈希值，映射到同一个桶时，发生哈希冲突。

# q
unordered_map的负载因子超过阈值时会发生什么？
# a
插入新元素后，如果当前负载因子大于 max_load_factor，会自动扩展桶数量（通常翻倍或扩至大于当前元素数的合适素数），然后所有已有元素重新分配到新桶（rehash）。扩容和 rehash 会导致所有 bucket iterator 失效，但普通 iterator 不失效。

# q
C++ 的 unordered_map 如何解决哈希冲突？链表过长时如何优化？
# a
默认使用链地址法，每个桶维护一个链表。C++11 后，当链表长度超过阈值（如 8），链表转换为红黑树，将最坏时间复杂度从 O(n) 降为 O(log n)。

# q
如何避免 unordered_map 频繁 rehash？
# a
通过 reserve(n) 预分配足够桶，或调整 max_load_factor()（如设为 0.75）提前扩容，减少运行时 rehash。

# q
如何设计一个高质量的 hash 算法？
# a
优先使用位运算；对字符串，每个位乘以一个小素数，然后多次用另一个大素数取模，典型如碳33算法；可以组合两个 hash 函数，先 hash1 再 hash2，或同时对 key 应用两个 hash 函数后取模。
