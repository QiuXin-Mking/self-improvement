# q
最长连续序列问题在输入数组长度为1时返回0而非1的典型根因是什么？
# a
典型根因是算法未正确处理边界条件，例如未将最长连续序列的初始长度设为至少为1（当数组非空时），或者循环逻辑跳过了单元素时的长度计算。对于 `nums = [0]`，算法可能初始化 `max_len = 0`，且因缺少第二个元素无法进入比较或更新逻辑，直接返回初始值 0，而期望正确结果为 1。

# q
如何通过日志或调试定位单元素场景的边界处理缺陷？
# a
在代码关键位置插入打印语句，记录 `max_len`、`current_len` 的初始值和每次变化。对 `nums = [0]`，若输出 0 且日志显示 `max_len` 始终保持 0，从未被更新为 1，即可定位到初始值或循环条件有误。应重点检查是否对空数组做了特殊返回，却未对非空数组保证长度至少为 1。

# q
解决此类边界条件问题的标准流程是什么？
# a
1. 识别边界用例（空数组、单元素、全重复元素等）；  
2. 编写单元测试覆盖这些用例，如 `assert longestConsecutive([0]) == 1`；  
3. 审查算法初始状态和循环不变式，确保当输入长度为 1 时也能正确计算连续长度；  
4. 修正代码，通常是在返回前将 `max_len` 初始化为 `Math.max(max_len, current_len)` 或在非空时设默认值为 1；  
5. 回归运行全部测试，确认修复有效且未引入新问题。
