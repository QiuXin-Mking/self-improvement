# q
什么是强一致性？其核心特性有哪些？
# a
强一致性要求所有节点在任何时刻对同一数据的读取操作均返回最新写入的值。核心特性包括：
1. 全局同步：写操作需等待所有副本确认后才返回成功。
2. 顺序性：操作按全局顺序执行，读写顺序严格一致。
3. 实时性：无延迟感知，数据状态始终统一。

# q
实现强一致性的常见方式有哪些？
# a
- 同步复制：写操作同步传播至所有副本（如传统关系型数据库的主从同步）。
- 共识算法：通过 Paxos、Raft 等协议协调节点间的操作顺序，确保一致性。
- 分布式事务：如两阶段提交（2PC）、三阶段提交（3PC），通过协调者管理事务的提交或回滚。

# q
什么是最终一致性？其核心特性有哪些？
# a
最终一致性允许数据在更新后存在短暂的不一致，但保证在没有新操作的情况下，所有副本最终会收敛到一致状态。核心特性：
1. 异步同步：写操作优先在本地完成，异步传播至其他副本。
2. 容忍延迟：允许副本间存在时间窗口内的不一致。
3. 最终收敛：通过重试、冲突解决等机制确保最终一致。

# q
实现最终一致性的常见方式有哪些？
# a
- 异步复制：如 Amazon Dynamo 的“最终一致性读写”模式。
- 版本控制：为数据附加时间戳或版本号，冲突时按规则（如“最后写入胜出”）合并。
- 读取修复（Read Repair）：读取时检测不一致并后台修复。

# q
强一致性与最终一致性的主要区别是什么？
# a
- 数据可见性：强一致性立即全局可见；最终一致性延迟后最终可见。
- 性能：强一致性低（同步阻塞）；最终一致性高（异步非阻塞）。
- 适用场景：强一致性用于金融、实时交易；最终一致性用于社交、缓存、高并发读写。
- 容错性：强一致性在网络分区时可能不可用；最终一致性网络分区后仍可用，但数据可能暂时不一致。
- 实现复杂度：强一致性高（需共识算法或分布式事务）；最终一致性中等（依赖异步机制和冲突解决）。
