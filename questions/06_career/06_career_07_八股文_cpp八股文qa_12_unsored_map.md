# q
hash冲突是如何触发的？
# a
哈希冲突是因哈希函数将不同键计算出相同哈希值导致。例如，字符串 "123" 和 "1234" 可能被简化哈希函数映射到同一桶。

# q
unordered_map的负载因子超过阈值时会发生什么？
# a
1. 插入新元素后，如果当前负载因子大于 max_load_factor，unordered_map 会自动扩展桶（bucket）数量，一般桶的总数会翻倍或扩大到一个大于当前元素数量的合适素数。
2. 所有已有元素全部重新分配到新的桶中（即 rehash），以减少哈希冲突，提高访问效率。
3. 扩容和 rehash 的过程可能会导致所有指向桶中元素的 bucket iterator 失效（普通的 iterator 到元素本身不会失效）。

# q
C++ 的 unordered_map 如何解决哈希冲突？链表过长时如何优化？
# a
unordered_map 默认使用链地址法，每个桶维护一个链表。C++11 后，当链表长度超过阈值（如 8），链表会转换为红黑树，将最坏时间复杂度从 O(n) 降为 O(log n)。

# q
如何避免 unordered_map 频繁 rehash？
# a
通过 reserve(n) 预分配足够桶，或调整 max_load_factor()（如设为 0.75）提前扩容，减少运行时 rehash。

# q
如何设计一个高质量的 hash 算法？
# a
1. hash 算法优先使用位运算。
2. 字符串一般每个位乘以一个小素数，然后多次用另外一个大素数取模，典型用碳 33 算法。
3. 组合两个 hash：先 hash 函数 1 的结果，再 hash 函数 2；或同时对一个 key 的两个 hash 函数结果再取模。
