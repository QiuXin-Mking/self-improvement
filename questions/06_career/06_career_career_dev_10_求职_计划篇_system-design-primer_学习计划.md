# q
在系统设计面试中，标准的四步回答流程是什么？
# a
1. 明确需求与约束（Use Cases, Constraints, Assumptions）  
2. 高层设计（High-Level Design）  
3. 核心组件设计（Core Components）  
4. 扩展与优化（Scale & Trade-offs）

# q
CAP 定理和 BASE 理论分别是什么？它们的核心区别在哪里？
# a
CAP 定理：一致性（Consistency）、可用性（Availability）、分区容错性（Partition Tolerance）三者最多只能同时满足两个。  
BASE 理论：基本可用（Basically Available）、软状态（Soft state）、最终一致性（Eventual consistency），通过牺牲强一致性来换取高可用和分区容错性。  
核心区别：CAP 是分布式系统中的硬约束理论，BASE 是实现高可用系统时的务实策略，最终一致性是其核心特征。

# q
为什么大多数对象存储系统（如 S3、COS）选择最终一致性而不是强一致性？
# a
对象存储系统需要跨多节点、多机房甚至跨区域部署，必须保证高可用和分区容错性。根据 CAP 定理，无法同时满足强一致性和高可用/分区容错，因此大多数对象存储选择最终一致性。这种模型允许在短时间内数据不一致，但最终在所有副本上达成一致，从而兼顾大规模扩展、高并发写入和系统可用性。

# q
设计一个对象存储系统时，元数据管理有哪些核心方案和权衡？
# a
- **中心化元数据**：如 HDFS NameNode，结构简单、查询快，但单点瓶颈和扩展性差。  
- **分布式元数据**：如 Ceph RGW 通过 CRUSH 算法和 RADOS 集群分散元数据，扩展性强，但实现复杂。  
- 需支持 Bucket/Object 的快速映射、多租户隔离，并解决元数据的一致性和可用性权衡。通常分布式方案更适合大规模对象存储。

# q
如何设计支持大文件上传和断点续传的对象存储系统？
# a
采用分块上传（Multipart Upload）机制：  
1. 客户端将大文件切割成固定大小的块。  
2. 各块可独立上传到存储节点，支持并发和失败重传，实现断点续传。  
3. 所有块上传完成后，通过元数据服务记录块顺序及对象组成。  
4. 合并请求完成后，对象对外可读，并支持清理未完成的碎片块以节省空间。
