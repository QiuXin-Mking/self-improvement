# q
什么是强一致性？它的核心特性和实现方式有哪些？
# a
强一致性要求所有节点在任何时刻对同一数据的读取操作均返回最新写入的值。其核心特性包括：全局同步（写操作需等待所有副本确认后才返回成功）、顺序性（操作按全局顺序执行）和实时性（数据状态始终统一）。实现方式主要有：同步复制（如传统关系型数据库的主从同步）、共识算法（Paxos、Raft等）以及分布式事务（两阶段提交2PC、三阶段提交3PC）。

# q
最终一致性是如何定义的？通常采用哪些机制来实现？
# a
最终一致性允许数据在更新后存在短暂的不一致，但保证在没有新操作的情况下，所有副本最终会收敛到一致状态。其核心特性是异步同步、容忍延迟和最终收敛。常用的实现机制包括：异步复制（如Amazon Dynamo的最终一致性读写）、版本控制（附加时间戳或版本号，冲突时采用“最后写入胜出”等规则合并）以及读取修复（Read Repair，在读取时检测不一致并后台修复）。

# q
强一致性和最终一致性在性能、适用场景和容错性上的主要区别是什么？
# a
• 性能：强一致性因同步阻塞机制导致低性能，最终一致性通过异步非阻塞实现高吞吐。  
• 适用场景：强一致性适用于金融交易、实时系统等对准确性要求极高的场景；最终一致性适用于社交网络、分布式缓存、NoSQL数据库等可容忍短暂不一致的高并发场景。  
• 容错性：强一致性在网络分区时可能因无法达成共识而阻塞服务，最终一致性在网络分区后仍可用，但数据可能暂时不一致。

# q
选择强一致性或最终一致性的依据是什么？
# a
需要根据业务需求权衡一致性级别、性能和可用性。例如，对准确性要求极高的金融系统应优先选择强一致性，而社交平台或电商促销等对延迟敏感且可容忍短暂不一致的场景可接受最终一致性。
