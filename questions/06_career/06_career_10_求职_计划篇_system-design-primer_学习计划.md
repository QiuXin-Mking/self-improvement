# q
系统设计面试的标准四步流程是什么？
# a
1. 明确需求与约束（Use Cases, Constraints, Assumptions）—— 主动向面试官澄清用户规模、读写比例、多租户要求、一致性要求、延迟要求、数据规模等。
2. 高层设计（High-Level Design）—— 画出抽象架构图，包含主要组件。
3. 核心组件设计（Core Components）—— 深入设计关键模块，如元数据服务、数据节点等。
4. 扩展与优化（Scale & Trade-offs）—— 讨论扩展方案（如多区域复制、版本控制）以及一致性、可用性、性能之间的权衡。

# q
设计对象存储系统时，对元数据服务的管理有哪些关键设计要点？
# a
- 结构：需维护 Bucket/Object 的映射关系，支持快速查询。
- 架构选择：中心化元数据管理 vs 分布式元数据管理，需权衡复杂度与一致性。
- 性能要求：低延迟查找。
- 可参考的已有方案：Ceph RGW 的元数据管理。

# q
为什么大多数对象存储系统更倾向于选择“最终一致性”而不是“强一致性”？
# a
主要考虑高可用性和性能：在分布式环境下强一致性会增加延迟、降低吞吐量，且在分区发生时只能选择一致性或可用性（CAP 定理）。对象存储多采用“最终一致性 + 多副本”策略，在保证数据可靠的同时，提供更好的性能与扩展能力，通过异步复制最终使所有副本达成一致。

# q
面向对象存储系统，支持大文件上传和断点续传的常见设计手段有哪些？
# a
采用分块上传：将大文件切分为多个块（分片），每个块独立上传，可并发写入提高效率；记录已上传块的状态，支持断点续传；所有块上传完成后，在元数据服务中合并为完整对象。

# q
根据学习计划，设计一个对象存储系统时高层架构通常应包含哪些核心模块？
# a
客户端 → API 网关（负载均衡） → 元数据服务（管理 Bucket/Object 映射） → 数据存储节点（存储实际对象）；并引入 CDN（加速访问）、缓存层（提升读取性能）和多区域复制（高可用和灾备）。
