# q
Go 的变量分配在栈上还是堆上？

# a
**由编译器决定**，不是由 var 或 new 决定的。编译器进行逃逸分析，判断变量是否在函数返回后还被引用。

简单规则：
- 函数内部只在本函数用 → 栈上
- 被外部指针引用（逃逸了） → 堆上

和 C++ 的根本区别：C++ 明确 new = 堆、局部变量 = 栈。Go 不让你决定，编译器自己分析。

# q
逃逸分析（escape analysis）是什么？什么情况下会逃逸？

# a
逃逸分析是编译器决定变量分配位置的过程。如果变量在函数返回后还能被访问，就"逃逸"到堆上。

常见逃逸场景：
1. 返回局部变量的指针
2. 局部变量被闭包引用
3. interface 类型装箱（赋值给 interface{} 可能逃逸）
4. 变量太大，超过栈限制
5. slice/map 动态扩容（底层数组逃逸到堆）

常见不逃逸场景：
1. 局部变量只在函数内使用
2. 切片 append 在 cap 范围内
3. 函数内联后变量作用域在调用者栈上

# q
怎么查看哪些变量逃逸到堆上了？

# a
```bash
go build -gcflags="-m"  # 打印逃逸分析结果
go build -gcflags="-m -m"  # 更详细
```

输出示例：
```
./main.go:5:2: moved to heap: x   // x 逃逸了
./main.go:8:13: ... argument does not escape  // 没逃逸
```

建议写代码时偶尔看一下，培养"逃逸感"。

# q
new 和 make 在内存分配上有什么区别？

# a
new(T): 分配内存，清零，返回 *T。逃逸分析决定在哪分配。

make(T): 只用于 slice/map/channel，初始化内部数据结构（底层数组、哈希桶、环形缓冲区），返回 T。

```go
p := new(int)          // *int，逃逸分析决定位置
s := make([]int, 0, 8) // 底层数组可能逃逸到堆（大容量），s header 在栈
```

关键认知：new 不代表分配在堆上！Go 的 new 只是一个"获取已清零内存"的途径。

# q
Go 的 GC 用的什么算法？有几个阶段？

# a
Go 使用**并发三色标记-清除**算法，主要三个阶段：

1. **Mark Setup（STW 暂停）**：开启写屏障，将根对象（全局变量、goroutine 栈）标为灰色
2. **Concurrent Mark（并发）**：从根出发做三色标记，和 mutator 并发执行
3. **Mark Termination（STW 暂停）**：结束标记，处理残留
4. **Concurrent Sweep（并发）**：回收白色对象

三色标记法：
- 白色：未访问（潜在垃圾）
- 灰色：已访问，但子对象未访问完
- 黑色：已访问完毕，不会回收

写屏障保证了并发期间不丢标记（不会错误回收黑色→白色引用）。

# q
Go 的 GC 触发条件？

# a
1. **内存到达阈值**：GOGC 参数控制（默认 100，即堆内存增长 100% 触发 GC）
2. **定时触发**：长时间没 GC（默认 2 分钟）
3. **手动触发**：runtime.GC()

GOGC=100 意思是：当堆内存增长到上次 GC 后存活内存的 2 倍时触发。设为 off 可关闭 GC（极少情况用）。

# q
GC 的 STW（Stop The World）有多久？

# a
Go 1.5+ 的并发 GC 让 STW 极短，通常在微秒级。

STW 只在两个阶段发生：Mark Setup（开启写屏障）和 Mark Termination（结束标记），不随堆大小增长。

但 GOGC 设太低会频繁 GC，累积的 STW 时间也不小。设太高会省 GC 但省不下内存。默认 100 是比较好的平衡。

# q
什么代码习惯能减少 GC 压力？

# a
1. **预分配 slice/map**：`make([]int, 0, 1000)` 避免多次扩容
2. **复用 []byte buffer**：用 sync.Pool 避免频繁分配
3. **用值类型而非指针**：减少堆分配（但不是绝对的——小型 struct 传值可能更快）
4. **减少 interface 装箱**：interface 装箱常伴随堆分配
5. **用 strings.Builder 代替 + 拼接**：减少临时字符串
6. **避免频繁创建小对象**：特别是循环里

# q
sync.Pool 是什么？怎么用？

# a
sync.Pool 是一个临时对象池，用于复用分配成本高的对象，减轻 GC 压力。

```go
var bufPool = sync.Pool{
    New: func() interface{} {
        return make([]byte, 1024)
    },
}

func process() {
    buf := bufPool.Get().([]byte)
    defer bufPool.Put(buf)
    // 用 buf 做事情...
}
```

特点：
- 线程安全，无锁获取（有 per-P 缓存）
- 不保证持久存储：GC 时会清空 Pool
- 适用于：高频创建/销毁的临时对象（buffer、连接）
- 不适用于：需要持久存储的、有状态的、必须归还的

# q
Go 的指针和 C++ 的引用/指针有什么区别？

# a
| | Go 指针 | C++ 指针 | C++ 引用 |
|---|---|---|---|
| 运算 | 禁止（unsafe 除外） | 自由加减 | 禁止 |
| 空值 | nil | nullptr | 不存在 |
| 多级指针 | 支持 | 支持 | 不支持 |
| GC | 参与 GC | 不管 | 不管 |
| 可重新赋值 | 是 | 是 | 否 |

Go 指针受 GC 管理，无需手动释放。但也不能像 C 那样用指针做数组运算。
