# q
线上NAS存储空间不足的典型根因是什么？
# a
通常是某些渲染器或DCC软件插件、缓存文件堆积导致占用过大。例如在该案例中，`/mnt/lustre/cyy/users/LHPlugins/` 下 `3dsmax`（923G）、`Maya`（623G）、`Cinema4d`（318G）、`V-Ray`（131G）等目录占用极高，远超正常插件体积，可能存在版本备份、临时渲染缓存或重复部署文件。

# q
如何快速定位NAS上哪些目录占用空间最大？
# a
登录NAS挂载点所在节点，执行：
```bash
du -sh /mnt/lustre/cyy/users/LHPlugins/*
```
该命令会输出每个一级子目录的总大小，按需进一步进入子目录排查。本次排查关键输出示例：
```
923G    ./3dsmax
16G     ./aces
318G    ./Cinema4d
623G    ./Maya
131G    ./V-Ray
...
226G    ./渲染服务工具脚本
```
根据结果确认异常膨胀的目录。

# q
解决NAS文件系统空间不足的标准流程是什么？
# a
1. 确认挂载点使用率与inode情况（`df -h`、`df -i`）
2. 使用`du -sh *`逐层定位大目录
3. 识别可清理内容（旧版本、临时文件、安装包、日志等）
4. 与相关用户确认后清理或移动至归档存储
5. 如有必要，调整配额或扩容

# q
在此案例的bat文件中可能包含哪些排查线索？
# a
检查了 `corona12update1_3dsmax2023.bat` 的内容（`cat` 命令），该bat文件可能是插件部署或环境初始化脚本。排查时需关注脚本是否触发反复下载、解压或复制操作，导致同一插件版本被多次部署。结合`du -sh`结果中 `Corona` 子目录大小异常，可推测存在未清理的历史版本或重复副本。
